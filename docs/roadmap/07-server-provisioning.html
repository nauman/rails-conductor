Server provisioning via provider APIs

Move Conductor from registering pre-existing boxes to creating and bootstrapping VMs directly through cloud-provider APIs, closing the biggest gap against Hatchbox's one-click server flow.

Pillar: Provisioning & Provider Automation Priority: P1 Effort: L Status: Planned

Problem

Conductor can register a server that already exists, but it cannot create one. Hatchbox's core onboarding flow is “pick a provider, pick a size, click — you get a hardened, provisioned box.” Conductor instead requires the operator to go to DigitalOcean/Hetzner/etc., spin up a VM by hand, copy its IP and SSH details back into Conductor, and only then run provisioning. The Provisioning & Provider Automation pillar audited at roughly 32% precisely because the “create the server” half is missing entirely. This is the single biggest friction point in standing up a new app on the fleet.

Current state

Server creation is fully manual. Server (app/models/server.rb) is a registration record: it declares PROVIDERS = %w[hetzner digitalocean linode vultr aws gcp azure] and validates that provider is one of them, but provider is purely descriptive metadata — the model stores name, ip_address, SSH details (ssh_user, ssh_port, ssh_key), and live metrics, and nothing in it talks to a provider API. The closest thing to provisioning is ProvisioningService (app/services/provisioning_service.rb), which takes a script_run, opens an SshConnection to an already-existing @server, and streams a Script body over SSH while logging output — i.e. it runs setup/hardening commands on a box that must already be reachable. There is a Hetzner spec at docs/plans/provisioning-hetzner.md and a bootstrap spec at docs/plans/server-bootstrap.md, plus related docs/plans/ssh-keys.md, but no provider-API code that calls out to create a VM. In short: Conductor has the “configure the box” half (SSH + Scripts) and none of the “summon the box” half.

Proposed design

Introduce provider adapters that wrap each cloud's compute API behind a uniform interface (create_server, destroy_server, list_servers, list_sizes/list_regions). Start with a HetznerCloudAdapter, then add DigitalOceanAdapter. A one-click “Create server” action runs a pipeline: call the adapter to create a VM with the fleet SSH key injected at boot, poll until the VM is running and its IP is assigned, create the corresponding Server record (capturing provider, ip_address, and the ssh_key used), then drive the existing ProvisioningService to run the standard hardening/bootstrap Scripts (server-harden, base bootstrap) against the new box. Destroy reverses it: call the adapter to delete the VM and reconcile the Server record. Provider tokens are sensitive, so they live in a per-organization encrypted credential model (mirroring how DatabaseCluster uses encrypts :admin_password) rather than in plaintext config. The whole flow is exposed both in the web UI and as an MCP tool so the GPT assistant can create infrastructure.

Implementation slices

  1. Provider-credential model. A per-org, encrypted store for provider API tokens (e.g. ProviderCredential with provider, encrypted token), scoped through organization like other resources.
  2. Hetzner adapter. HetznerCloudAdapter implementing create_server (with SSH key injection), destroy_server, and list_servers/sizes/regions against the Hetzner Cloud API, with retry/poll-until-ready logic.
  3. Create→register→bootstrap pipeline. An orchestrating service that creates the VM, waits for boot + IP, persists a Server with the fleet ssh_key, then invokes ProvisioningService for the existing hardening/bootstrap Scripts — surfacing live progress like the current provisioning logs.
  4. DigitalOcean adapter + UI/MCP. Add DigitalOceanAdapter behind the same interface, a “Create server” UI flow (provider/region/size selection), and a create_server_from_provider MCP tool mirroring the existing tool patterns.

Acceptance criteria

  • A single UI action (and the create_server_from_provider MCP tool) creates a Hetzner Cloud VM, registers it as a Server with provider, IP, and SSH key populated, and runs the existing hardening Scripts against it — with streamed progress.
  • The created server is immediately reachable over SSH using the injected fleet key and reports metrics like any registered server.
  • A destroy action deletes the underlying VM via the provider API and reconciles/removes the Server record.
  • DigitalOcean works through the same interface and UI/tool.
  • Provider API tokens are stored encrypted per organization and are never written to logs or plaintext config.

Dependencies & related

Consumes the existing SSH + Scripts machinery: ProvisioningService (app/services/provisioning_service.rb), SshConnection, and the Server registration model (app/models/server.rb, including its PROVIDERS list). Requires the fleet SSH key plumbing in docs/plans/ssh-keys.md and the hardening/bootstrap definitions in docs/plans/server-bootstrap.md. Directly advances the Hetzner spec at docs/plans/provisioning-hetzner.md. Newly created servers are the substrate for database clusters and accessories, so it precedes/enables that data-layer work (docs/plans/database-clusters.md). The encrypted-credential pattern can later be reused for DNS/Cloudflare tokens (docs/plans/cloudflare.md, docs/plans/domains-dns.md).
