Caddy management console + per-app multi-subdomain

A full Caddy surface in Conductor: see and edit every route and cert across the fleet, and give an app its own per-app Caddy API access so it can register/retire subdomains on the fly (e.g. calm.page handing out tenant subdomains via JSON).

Pillar: Routing & edge Priority: P1 Effort: M Status: Planned

Problem

CaddyClient can upsert/remove a single route, but there's no console — no place to see all routes, certs, and reachability across servers, or to diff the live config against what Conductor expects. And there's no per-app path: an app like calm.page that needs to mint subdomains at runtime has no scoped, safe way to drive Caddy itself. Slot 18 picks the proxy mode at deploy; this is the ongoing management + dynamic-routing surface.

Current state

CaddyClient (Admin API on :2019) does route CRUD over SSH; conductor_domain (add/remove) is the MCP tool. Routes aren't modelled/persisted in Conductor, there's no cert/drift view, and there's no per-app credential to let an app call Caddy in a bounded way.

Proposed design

Console: a Route model synced from each server's live Caddy config, a fleet routing view (domain → app/upstream, cert status, last-seen), and drift detection (live config vs expected) with one-click reconcile. Per-app Caddy API: issue an app a scoped routing credential + a narrow Conductor-mediated endpoint, so the app POSTs a JSON payload ({subdomain, upstream}) and Conductor upserts the Caddy route on its behalf — the app never touches the raw Admin API, every change is org-scoped and audited. Wildcard-cert aware for *.app.com tenant subdomains.

Implementation slices

  1. Route model + sync from live Caddy; fleet routing console (domains, upstreams, cert status) with drift detection + reconcile.
  2. Per-app routing credential (scoped) + a Conductor-mediated POST /apps/:id/subdomains (JSON) that upserts a Caddy route, org-scoped + audited.
  3. Multi-subdomain UX: list/add/remove an app's subdomains; wildcard-cert handling for *.domain.
  4. MCP parity: conductor_domain gains subdomain + console actions; agents manage routing the same way the app does.

Acceptance criteria

  • One console shows every Caddy route + cert across the fleet, flags drift, and reconciles with one click.
  • An app (e.g. calm.page) registers/retires a subdomain by POSTing a JSON payload to its scoped endpoint, and it routes immediately — without touching the raw Admin API.
  • Every subdomain change is org-scoped, audited, and visible in the console; revoking the app's credential stops it.

Dependencies & related

Extends CaddyClient + the per-app proxy mode (caddy mode is the prerequisite). Cert/wildcard work touches domains & DNS. Core of the Routing & edge pillar.
