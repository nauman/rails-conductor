Seed management & idempotency check

Run an app's seeds as part of deploy (opt-in), and verify those seeds are idempotent — flag any that create duplicates or fail on re-run.

Pillar: Runtime Backends Priority: P1 Effort: S Status: Planned

Problem

A Conductor deploy runs db:prepare on boot (migrations), but never db:seed. For an app whose runtime behaviour depends on seeded rows — reference data, a demo/tenant account, role users — the app deploys "successfully" and then breaks at runtime with no signal. This bit us live: a deployed app's role-based demo login 500'd because the demo account it looks up is only created by seeds, which had never run. The operator has no UI affordance to run seeds, and no way to know seeds were needed.

The second half of the problem is safety: seeds are only safe to run repeatedly if they're idempotent (find_or_create_by, upserts). Many real seed files mix idempotent reference data with non-idempotent demo generators that insert duplicate rows every run. Running such a seed on every deploy would silently bloat the database.

Current state

The Kamal entrypoint runs ./bin/rails db:prepare only; seeds are out of band. Conductor has no concept of an app's seed step, no per-app toggle, and no exec path into the app container (see in-container task runner). Today running seeds means an operator dropping to a terminal and remembering kamal app exec --reuse "… bin/rails db:seed" with the right env flags.

Proposed design

Two coupled features:

  • Run-seeds-on-deploy toggle — a per-app boolean (with optional env, e.g. SEED_DEMO=1) that, when set, runs db:seed as a post-migrate deploy step via the task runner, with the output streamed into the deployment log.
  • Idempotency check — a one-click (and pre-deploy) verification that runs the seed twice inside a throwaway/transactional context and diffs row counts per table. If the second run changes any counts (or errors), the seed is flagged non-idempotent with the offending tables, and on-deploy seeding is blocked until acknowledged.

The check leans on the same in-container exec used for the task runner, so it runs against the app's real schema and seed code, not a guess.

Implementation slices

  1. Per-app run_seeds_on_deploy flag + optional seed env; surface it on the app page and via MCP/update_app.
  2. Deploy step: after migrations, if enabled, invoke the task runner's db:seed and stream output to the deployment log.
  3. Idempotency probe: run seeds twice in a transaction (rolled back), capture per-table row-count deltas, return a pass/fail report with offending tables.
  4. UI + gate: show the idempotency verdict on the app page; block on-deploy seeding for non-idempotent seeds until the operator overrides.

Acceptance criteria

  • An app can be configured to run its seeds on every deploy; the seed output appears in the deployment log.
  • Conductor can report whether an app's seeds are idempotent, naming any tables that change on a second run.
  • A non-idempotent seed cannot be silently attached to on-deploy seeding without an explicit override.
  • Running seeds is possible from the UI/MCP without an operator hand-crafting a kamal app exec command.

Dependencies & related

Built on the in-container task runner (executes db:seed against the running app) and the Kamal control machine (deploy pipeline that gains the post-migrate seed step). Related: deploy hooks.
