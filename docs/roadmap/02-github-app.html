GitHub App & deploy keys (private repo access)

Give Conductor real git credentials — a GitHub App or per-repo deploy keys — so it can clone private repos wherever the checkout happens.

Pillar: Provisioning & Provider Automation Priority: P0 Effort: M Status: Planned

Problem

Deploying a private repo requires git authentication at the point where the code is checked out — and Conductor has none. The redesigned KamalDeployer clones inside Conductor's own container; NativeDeployer clones on the target box. Neither location has credentials for a private repo today (and we've proven the fleet box has no git auth configured at all). Public repos work; the moment a customer has a private app, every deploy fails at git clone.

Current state

App#repository_url (app/models/app.rb) is a plain string with no associated credential. Both deployers run an unauthenticated clone:

  • KamalDeployer (app/services/kamal_deployer.rb, sync_repo_command) builds git clone --branch <branch> <repository_url> <checkout_dir> on first deploy and git fetch origin && git reset --hard origin/<branch> thereafter, run via LocalShell inside Conductor's container. No token, no SSH key for git — the header comment even flags this: "the app repo reachable (public, or a deploy key/token — separate backlog item)."
  • NativeDeployer (app/services/native_deployer.rb) exports REPO_URL / BRANCH into the environment (env_exports) and hands them to the built-in app-setup / app-deploy shell scripts run over SSH on the box. Those scripts clone with whatever git auth exists on the box — which is none.

The only SSH key Conductor manages is the server's key (app.server.ssh_key.private_key, materialized by KamalDeployer#write_ssh_key for net-ssh to reach the host) — that's for connecting to the box, not for authenticating to GitHub. There is no model, field, or flow for a git credential anywhere.

Proposed design

Introduce a git-credential layer with two interchangeable strategies, injected into the clone step in both deployers:

  • Deploy key (per repo) — Conductor generates an SSH keypair, stores the private half, and the user adds the public half as a read-only deploy key on the repo. The clone uses an SSH URL with GIT_SSH_COMMAND="ssh -i <key>".
  • GitHub App (per org) — Conductor is installed as a GitHub App; per clone it mints a short-lived installation access token and clones over HTTPS via GIT_ASKPASS (or x-access-token:<token>@github.com). This also unlocks the shared webhook secret and commit-status / checks API used by auto-deploy.
# KamalDeployer — clone with an injected credential (deploy-key variant)
def sync_repo_env
  return {} unless (key = app.git_deploy_key&.private_key_path)
  { "GIT_SSH_COMMAND" => "ssh -i #{key} -o IdentitiesOnly=yes" }
end

# NativeDeployer — credential reaches the box as a scoped token, not a stored secret
exports << "export GIT_ASKPASS=#{askpass_helper}"  # echoes a short-lived token

For the GitHub App path, tokens are minted at deploy time and expire (~1h), so nothing long-lived sits on the box; deploy keys are revocable from the repo settings and scoped to a single repo.

Implementation slices

  1. Deploy-key model + generation: a GitDeployKey belonging to an App (or repo), with keypair generation and exposure of the public key in the UI for the user to paste into GitHub.
  2. Inject into clone: thread the credential into both deploy paths — GIT_SSH_COMMAND in KamalDeployer#sync_repo_command (local clone) and a GIT_ASKPASS/key export in NativeDeployer#env_exports so the on-box scripts can authenticate.
  3. GitHub App OAuth/installation flow: register the App, handle the installation callback, store installation id per org, and mint short-lived installation tokens on demand.
  4. Commit status / checks: post deploy start/success/failure back to the commit via the App token, tying Deployment outcomes to the GitHub UI.

Acceptance criteria

  • An app pointing at a private repo deploys successfully through both NativeDeployer and KamalDeployer.
  • GitHub App installation tokens are short-lived and scoped to the installed repos; no long-lived token is written to the target box.
  • Revoking the deploy key (or uninstalling the App) causes subsequent clones to fail cleanly with a clear error in the deployment log.
  • The credential is never logged or echoed into the Deployment#log stream.

Dependencies & related

Direct prerequisite for Auto-deploy on git push — the GitHub App that authorizes clones is also the natural source of the webhook delivery and signing secret, and enables commit-status updates once auto-deploy is live. Touches the clone steps in app/services/kamal_deployer.rb and app/services/native_deployer.rb, and adds a credential association to app/models/app.rb. Distinct from the existing server SSH key (Server#ssh_key), which authenticates to the box, not to GitHub.
