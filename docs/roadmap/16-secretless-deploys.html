Secretless deploys (vault-resolved secrets)

Resolve a deploy's secrets from a vault at deploy time instead of storing them as plaintext env vars — so an agent can trigger a deploy without ever reading a secret value, and secrets stay encrypted at rest.

Pillar: Agent-native Priority: P1 Effort: M Status: Planned

Problem

KamalDeployer builds .kamal/secrets from the app's EnvVariables, which live (decryptably) in Conductor's DB. That forces an uncomfortable trade-off for agent-driven deploys: either every production secret is pre-entered as a plaintext env var, or an agent must read the secrets to set them — and an agent reading prod secrets is a trust-boundary violation. This surfaced concretely on 2026-06-21: an MCP-driven self-deploy failed at docker login -p because KAMAL_REGISTRY_PASSWORD was absent, and the safe path (fetch it from the vault) was correctly blocked because the agent shouldn't see it. Duplicating every secret as a DB env var also widens blast radius.

Current state

KamalEnvWriter.secrets_content(app) writes .kamal/secrets from app.env_variables. Pre-flight validation (shipped 2026-06-21) verifies the required keys a repo's deploy.yml declares are present — but it still relies on plaintext values being stored. The canonical secrets already live encrypted in a vault (localvault, cloud-synced); Conductor has no integration with it. Kamal natively supports secret helpers ($(command) in .kamal/secrets), which is the clean hook.

Proposed design

Introduce a pluggable SecretSource (vault-provider) interface. An app/org references secrets by name from a provider instead of (or alongside) plaintext EnvVariables. At deploy, KamalEnvWriter resolves the referenced keys from the provider using a single Conductor-held, scoped vault token — or emits a .kamal/secrets that shells out to the vault via kamal's secret helpers. The agent triggers the deploy and passes/reads zero secret values; the trust anchor narrows from "N plaintext secrets in the DB" to "one revocable vault token." localvault is one provider behind the interface (kept generic in the public repo, not hardcoded).

Implementation slices

  1. SecretSource interface + a vault-backed secret reference (key name + provider) that renders into .kamal/secrets at deploy, never persisted plaintext.
  2. A localvault provider and a generic provider (HTTP/CLI) behind the interface; one scoped vault token per org, stored as a Credential.
  3. Resolve-at-deploy in KamalEnvWriter (or emit kamal $(...) secret helpers); extend pre-flight to check key reachability without reading values.
  4. Secretless MCP deploy: the deploy tool works with no secret values passed or read; the audit log records "resolved from vault: KEY" (never the value).

Acceptance criteria

  • A kamal deploy succeeds with all secrets resolved from the vault; no secret is stored as a plaintext EnvVariable.
  • An MCP agent triggers the deploy without reading any secret value (and the harness need not grant it secret access).
  • Pre-flight reports a missing/unreachable vault key by name, never exposing the value.
  • The provider is pluggable; localvault is one implementation, swappable for a cloud vault.

Dependencies & related

Builds directly on pre-flight secret validation (shipped) and KamalEnvWriter + the Kamal control machine. Strongly complements multi-tenant MCP — secretless deploys are what make "anyone (incl. an agent) can deploy" safe.
