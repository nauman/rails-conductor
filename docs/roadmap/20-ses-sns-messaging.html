AWS SES + SNS messaging (email + SMS)

One place to monitor and manage transactional email (SES) and SMS (SNS) for the fleet — sending health, reputation, identities — from UI and over MCP.

Pillar: Provider automation Priority: P1 Effort: M Status: Planned

Problem

Apps send mail through SES (SMTP creds injected at deploy), but Conductor has no visibility into deliverability — bounce/complaint rates, reputation, verified identities — and no way to manage them. And there's no SMS path at all. Email/SMS health is operational state the control plane should own.

Current state

SES SMTP send works (credentials via env/vault). No SES API client, no stats, no identity management, no SNS. Note: SES is email-only — SMS is AWS SNS (or Pinpoint), a separate service.

Proposed design

An AWS client (SES + SNS) using IAM creds from the secret source. Monitor: SES sending stats (sends, bounces, complaints, reputation), verified identities/domains; SNS delivery + spend. Manage: verify identities, manage config sets, send test email/SMS, SNS topics. UI panel + MCP conductor_messaging tool. Alerts when bounce/complaint rates cross thresholds (feeds Continuous maintenance).

Implementation slices

  1. AWS client + credentials (IAM via secret source); SES monitor: sending stats, bounce/complaint, identities.
  2. SES manage: verify domain/email identities, config sets, send test.
  3. SNS (SMS): send, delivery status, spend monitor — clearly separate from SES.
  4. MCP conductor_messaging tool + threshold alerts on reputation/bounce rate.

Acceptance criteria

  • From UI and MCP, an operator sees SES sending health (bounce/complaint/reputation) and verified identities.
  • SMS can be sent via SNS with delivery + spend visible; the SES/SNS distinction is explicit in the UI.
  • An alert fires when bounce or complaint rate crosses a configured threshold.

Dependencies & related

Credentials via the secret source; alerts feed alerts & notifications. Part of the Connected Services hub.
