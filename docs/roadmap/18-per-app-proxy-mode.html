Per-app proxy mode (Caddy or kamal-proxy)

Let each app choose how it's fronted — Caddy (dynamic JSON routes via the admin API, the fleet default) or kamal-proxy (Kamal's built-in). Caddy mode unlocks on-the-fly subdomains (e.g. calm.page handing out tenant subdomains via a JSON payload).

Pillar: Fleet Control Priority: P1 Effort: M Status: Planned

Problem

The fleet's intended router is Caddy (VISION.md: "Caddy — reverse proxy + Let's Encrypt + Admin API on :2019"), and Conductor already drives it with JSON route config (CaddyClient#upsert_route). But each app's routing is dictated by its own deploy.yml: a Kamal app defaults to kamal-proxy, which contends with Caddy for :80/:443 and can't do dynamic per-tenant subdomains. Conductor's own deploy.yml still carries a kamal proxy: block — the outlier. There's no way to say "this app is fronted by Caddy" vs "by kamal-proxy", and no clean path for an app like calm.page to register a subdomain at runtime.

Current state

CaddyClient manages routes over Caddy's admin API (JSON: domain → upstream), exposed via the conductor_domain (add/remove) MCP tool. App has port and domain but no proxy-mode field. Conductor deploys an app by cloning its repo and running kamal deploy with the app's committed deploy.yml (proxy block as-is). Kamal and Caddy compose fine — set proxy: false so Kamal skips kamal-proxy, publish the container's port, and let Caddy reverse-proxy to it — but Conductor doesn't orchestrate that today.

Proposed design

Add App#proxy_mode (caddy | kamal; default caddy for this fleet). At deploy, the deployer honors it:

  • caddy — override the cloned deploy.yml to proxy: false, ensure the container publishes its port, and CaddyClient#upsert_route the domain → 127.0.0.1:<port> (or a unix socket). Subdomains become a JSON upsert away.
  • kamal — leave the app's kamal-proxy block untouched (default Kamal behavior).

Same override mechanism Conductor already uses for .kamal/secrets. Kamal still builds/boots the container; Caddy just fronts it.

Implementation slices

  1. App#proxy_mode column (caddy|kamal) + a toggle on the app form.
  2. Deployer honors the mode: caddy → render proxy: false + publish port + upsert the Caddy route on deploy; kamal → unchanged.
  3. Move Conductor itself + calm.page to caddy mode — after confirming the live box's :443 binding so routing isn't cut.
  4. calm.page dynamic subdomains: a JSON endpoint / MCP tool that upserts a Caddy route per tenant subdomain (or a wildcard) via CaddyClient.

Acceptance criteria

  • An app can be set to Caddy or kamal-proxy in the UI; the next deploy honors it.
  • A caddy-mode app is reachable via a Caddy route with no kamal-proxy port contention.
  • calm.page can add a subdomain via a JSON payload and it routes immediately (no redeploy).
  • kamal-mode apps deploy exactly as before — no regression.

Dependencies & related

Builds on CaddyClient + the conductor_domain tool (present) and the Kamal control machine deploy path. Flipping Conductor's own proxy is also the change that removes the proxy risk from a self-deploy. Related: domains & DNS.
