Live app log streaming in the UI

Tail a running managed app's logs in the browser — docker/kamal container logs or native journald — the way Hatchbox surfaces app logs, reusing Conductor's existing ActionCable streaming plumbing.

Pillar: Fleet Control Priority: P1 Effort: M Status: Planned

Problem

Conductor already streams deploy and script output live, but once an app is running there is no way to see what it is doing. To read a managed app's logs you have to SSH to the box and run kamal logs or docker logs / journalctl by hand. Hatchbox shows live app logs in its UI; Conductor has no app-log tailing surface at all.

Current state

Conductor has a working browser-streaming pattern, but only for ephemeral run output. Script runs stream over an ActionCable channel keyed by id: ScriptRun#append_log persists each line and broadcasts it via ActionCable.server.broadcast("script_run_#{id}", { line: line }), with start!/finish! broadcasting status transitions (app/models/script_run.rb:21,26,32). The job side runs the command through SshConnection#execute_stream, which opens a channel and yields :stdout/:stderr chunks (app/services/ssh_connection.rb). Deployers broadcast the same way: app_deployer.rb:53,79, native_deployer.rb:111,125, and kamal_deployer.rb:136,150 all call ActionCable.server.broadcast as deploy logs accumulate (with lines persisted via Deployment#append_log).

For live app state, Conductor relies on SSH-backed docker commands in ContainerStatus (app/services/container_status.rb): docker/native apps are inspected by container name (docker inspect --format '{{json .State}}' <container>), and kamal apps are detected by the label=service=<kamal_service> filter on docker ps. Native apps run under systemd (the systemd-setup unit seeded in db/seeds.rb). Crucially, all of this is point-in-time status — there is no app-log tailing endpoint. kamal logs / docker logs exist only via the CLI on the box, never piped to the browser.

Proposed design

Add an SSH-backed follow stream that runs a long-lived tail command on the remote host and pushes lines to the browser over a dedicated ActionCable channel, reusing the exact execute_stream + broadcast pattern already used for scripts and deploys. The remote command is chosen per deploy method: for docker/kamal apps, docker logs -f against the container resolved by the same label/name logic ContainerStatus already uses (label=service=<kamal_service> for kamal, container name for native-docker); for native systemd apps, journalctl -fu <unit>. The UI gets a per-app Logs tab with a follow toggle and a bounded in-browser buffer so memory stays flat, plus a recent-N preload.

Implementation slices

  1. LogStream job + channel. A LogStreamJob (or stream-scoped channel) runs a follow command through SshConnection#execute_stream and broadcasts each line on an app_logs_<app_id> ActionCable channel, mirroring ScriptRun#append_log; the follow process is torn down when the subscriber unsubscribes.
  2. Per-deploy-method log source. Resolve the right command per app: docker logs -f by container name/label for docker & kamal (reusing ContainerStatus's resolution), journalctl -fu <unit> for native systemd.
  3. UI Logs tab. Add a Logs tab to the app show page with a follow start/stop toggle that subscribes/unsubscribes from the channel.
  4. Recent-N + search. Preload the last N lines (--tail / -n) on open and add a client-side filter over the buffered lines.

Acceptance criteria

  • Opening an app's Logs tab shows a live tail for docker, kamal, and native (systemd/journald) apps.
  • The user can stop and restart following without reloading; stopping ends the remote follow process.
  • The browser buffer is bounded (recent-N) so a long-lived tail does not grow memory without limit.
  • Streaming reuses the existing ActionCable + execute_stream plumbing rather than a new transport.

Dependencies & related

Reuses SshConnection#execute_stream (app/services/ssh_connection.rb), the ActionCable broadcast pattern from ScriptRun (app/models/script_run.rb) and the deployers (app_deployer.rb, native_deployer.rb, kamal_deployer.rb), and the container/unit resolution logic in ContainerStatus (app/services/container_status.rb). Related to the background worker plan (plan-background-workers.html), whose workers will want their own per-process log source.
