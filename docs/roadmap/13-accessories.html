Redis & MySQL accessories

Extend Conductor's shared-cluster provisioning beyond PostgreSQL to the Redis and MySQL accessories that most Rails apps depend on, matching Hatchbox's data offering.

Pillar: Data & Backups Priority: P2 Effort: M Status: Planned

Problem

Conductor can provision a per-app PostgreSQL database on a shared cluster, but Hatchbox also offers Redis (for Rails cache and Sidekiq queues) and MySQL. The overwhelming majority of Rails apps need Redis for caching and background jobs, and a meaningful share still run on MySQL. Without these, an operator who moves a Rails app onto Conductor still has to hand-provision its cache and queue store outside the platform, which breaks the “UI-driven fleet control plane” promise.

Current state

The data layer is Postgres-only and hard-wired to one engine. DatabaseCluster (app/models/database_cluster.rb) belongs to a Server, encrypts an admin_password, and exposes provision_database!, which generates a 24-byte password, creates a pending Database record, and delegates to PostgresClusterClient. PostgresClusterClient (app/services/postgres_cluster_client.rb) runs admin SQL over SSH by base64-piping a statement into docker exec -i <container> psql -U <admin> -d postgres; create_database issues CREATE ROLE … LOGIN PASSWORD … CREATEDB followed by CREATE DATABASE … OWNER …, and drop_database issues the matching DROP DATABASE/DROP ROLE. The resulting Database (app/models/database.rb) builds a fixed postgres:// connection URL from username, password, the cluster's container_name, port, and name. The MCP surface is equally Postgres-specific: RegisterDatabaseClusterTool (app/tools/register_database_cluster_tool.rb) registers “a running postgres container”, and ProvisionDatabaseTool (app/tools/provision_database_tool.rb) is described as “Provision a Postgres database” and rescues PostgresClusterClient::Error. The web UI (app/controllers/database_clusters_controller.rb) defaults port to 5432 on new clusters. There is no Redis or MySQL client, model field, or tool anywhere in the codebase, and no engine concept to distinguish them. See also docs/plans/database-clusters.md and docs/plans/data-layer.md.

Proposed design

Generalize the cluster/accessory concept around an engine attribute (postgres|mysql|redis) on DatabaseCluster (or a renamed Accessory/Cluster), and introduce a per-engine client behind a common interface (create_database/drop_database, or more neutrally provision/deprovision). The existing PostgresClusterClient becomes the postgres implementation. MysqlClusterClient mirrors it closely — docker exec -i <container> mysql running CREATE DATABASE, CREATE USER … IDENTIFIED BY …, and GRANT ALL ON db.* TO user, with matching DROP on deprovision — and yields a mysql2:// URL. RedisAccessoryClient is conceptually different: rather than per-app databases it runs one shared Redis container per cluster and hands out an isolated logical handle, either a numbered logical DB index (redis://host:6379/3) or, preferably, a key-namespaced URL/ACL user so apps don't collide on the 16-DB limit. Database (renamed Accessory where natural) gains an engine-aware connection_url so each engine emits the right scheme. The provisioning UI and the two MCP tools accept and branch on engine so a single “Add accessory” flow covers all three.

Implementation slices

  1. Engine column + client interface. Add engine (default postgres) to clusters/accessories, define a thin client interface with a registry that maps engine→client, and refactor PostgresClusterClient to implement it. Make provision_database! resolve its client by engine instead of hard-coding PostgresClusterClient.
  2. Redis accessory. Register a shared Redis container as a cluster; RedisAccessoryClient allocates a per-app logical DB index (or namespaced ACL user) and surfaces a redis:// URL. Track the allocated index on the accessory record so allocation/release is idempotent.
  3. MySQL client. Implement MysqlClusterClient (CREATE DATABASE/CREATE USER/GRANT, and the inverse on drop) over the same docker exec SSH path, emitting a mysql2:// URL.
  4. Generalize tools + UI. Teach RegisterDatabaseClusterTool and ProvisionDatabaseTool (and the clusters controller/views) about engine: per-engine defaults (port 5432/3306/6379), engine-aware copy, and a unified rescue across the client error types.

Acceptance criteria

  • An operator can register a Redis cluster and provision a Redis handle for an app via both the web UI and the provision_database MCP tool, receiving a working redis:// URL scoped to that app.
  • An operator can register a MySQL cluster and provision a MySQL database + user for an app, receiving a working mysql2:// connection string.
  • The PostgreSQL flow is unchanged in behavior and still emits a postgres:// URL.
  • Each engine surfaces its connection string on the accessory record and in the MCP/tool result payload.
  • Deprovisioning (drop) cleans up the underlying resource: dropped MySQL db/user, released Redis index/namespace, and the accessory record is removed or marked accordingly.

Dependencies & related

Builds directly on the existing cluster pipeline (app/models/database_cluster.rb, app/services/postgres_cluster_client.rb, app/tools/{register_database_cluster,provision_database}_tool.rb). Connection URLs feed app deploy config, so it pairs with deployment/Kamal work (docs/plans/deployment-kamal.md). Redis/MySQL containers themselves are typically stood up by server bootstrap/provisioning (docs/plans/server-bootstrap.md), and backups should eventually cover the new engines (docs/plans/backups-r2.md, docs/plans/postgres-restore.md). See docs/plans/database-clusters.md and docs/plans/data-layer.md for the data-layer baseline.
