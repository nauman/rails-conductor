MCP wire-protocol transport

Make Conductor's MCP server speak the real Model Context Protocol so Claude Desktop, Cursor, and any MCP client connect natively — not just shell-capable agents.

Pillar: Agent-native Priority: P1 Effort: M Status: Planned

Problem

Conductor exposes a custom HTTP API at /mcp/list + /mcp/call ({name, input} → {result}). It's perfect for a shell-capable agent (Claude Code) hitting it with curl, and that's how every deploy so far has run. But it is not the MCP wire protocol — no JSON-RPC 2.0 envelope, no initialize / tools/list / tools/call methods, no SSE / streamable-HTTP transport. So Claude Desktop and Cursor can't connect to it natively as an MCP server; "anyone can deploy from their agent" only holds for agents that can run a shell.

Current state

Mcp::ServerController (app/controllers/mcp/server_controller.rb) renders plain JSON for list/call/skill, with per-user/per-org bearer auth (slot 14) and a redacted audit log (McpCall). The tool set + schemas already come from ToolRegistry, and the judgment layer is served at /mcp/skill. Everything needed is present except the protocol envelope and transport.

Proposed design

Add a spec-compliant MCP endpoint alongside the existing custom API (keep the latter for curl/CI). Implement the Streamable HTTP transport: a single POST /mcp accepting JSON-RPC 2.0, handling initialize, tools/list (map ToolRegistry.definitions → MCP tool descriptors), and tools/call (dispatch to ToolRegistry.call, wrap the result as MCP content), with SSE for streaming/notifications. Reuse the slot-14 bearer auth + org scoping unchanged. Expose the skill via an MCP resource (or a prompt) so connected clients pull it automatically.

Implementation slices

  1. JSON-RPC 2.0 envelope + initialize handshake (capabilities, serverInfo).
  2. tools/list from ToolRegistry.definitions; tools/call → ToolRegistry.call, result wrapped as MCP content; errors as JSON-RPC errors.
  3. Streamable-HTTP transport (POST + SSE) at POST /mcp; reuse bearer auth + Current org/scope.
  4. Expose /mcp/skill as an MCP resource/prompt; document the one-line Claude Desktop / Cursor connect config in the MCP guide.

Acceptance criteria

  • Claude Desktop / Cursor connect to Conductor with a per-user token and see the tools natively (no shell, no curl).
  • tools/list matches ToolRegistry; tools/call deploys an app end-to-end via a real MCP client.
  • Org scoping + read-only scope (slot 14) apply identically over the new transport.
  • The existing /mcp/call custom API keeps working for curl/CI.

Dependencies & related

Builds directly on slot 14 (per-user/org tokens + scoping) and ToolRegistry//mcp/skill. Completes the agent-native pillar: any MCP client, not just shell agents. See the MCP guide.
