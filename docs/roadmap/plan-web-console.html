Web console (rails console / shell)

In-browser interactive terminal for an app — run rails console, kamal app exec, a host shell, or docker exec without leaving Conductor.

Pillar: Fleet Control Priority: P2 Effort: M Status: Planned

Problem

To open a Rails console, run a one-off shell command, or poke at a running container, an operator today has to drop to a terminal and remember the right incantation (kamal console, kamal app exec, docker exec -it …, or an SSH session into the box). Hatchbox offers in-browser console access as a first-class feature; Conductor has no interactive shell at all, which breaks the "manage the fleet entirely from the UI" promise the moment something needs hands-on debugging.

Current state

Conductor can already reach every box over SSH, but only in a one-shot, non-interactive way. SshConnection (app/services/ssh_connection.rb) opens a fresh Net::SSH.start per call with non_interactive: true and exposes three modes: execute (ssh.exec!, returns a single string), execute_with_status (captures stdout/stderr/exit code), and execute_stream (lines 100–135, opens a channel and yields [:stdout|:stderr, data] chunks one-way). None of these allocate a PTY or accept input back from the client — they run a command and stream output until it exits.

Real-time output today is one-directional over ActionCable: deploy logs are pushed from the deployers (app/services/app_deployer.rb, kamal_deployer.rb, native_deployer.rb) and script output from ScriptRun (app/models/script_run.rb) via ActionCable.server.broadcast(...), consumed by ActionCable.createConsumer() in app/views/deployments/show.html.erb and app/views/script_runs/show.html.erb. The cable endpoint is already mounted (config/routes.rb:138, mount ActionCable.server => '/cable'), and the channel base classes exist under app/channels/application_cable/ — but there is no console route, no interactive channel, and no terminal UI. Apps carry a deploy_method of docker / native / kamal (App::DEPLOY_METHODS in app/models/app.rb) and helpers like container_name (conductor-<slug>), kamal_service, and app_dir that a console bridge can target.

Proposed design

Add a browser terminal backed by xterm.js that talks to a new ActionCable channel, which in turn bridges to an interactive SSH session running a PTY on the target box. Unlike SshConnection's fire-and-forget channels, the bridge keeps the SSH channel open, requests a PTY (ch.request_pty), forwards keystrokes from the browser to the channel's stdin, and streams the channel's output back to xterm. The exact remote command is chosen per deploy_method: kamal app exec --interactive --reuse "bin/rails console" (or a plain shell) for Kamal apps, docker exec -it <container_name> … for docker/native containers, and a raw login shell on the host for box-level work. Access is locked to webmaster/owner roles only, every command (or the full session transcript) is persisted to an audit log modeled on the existing McpCall record (app/models/mcp_call.rb), and idle sessions are torn down by a timeout.

Implementation slices

  1. PTY bridge: a new ActionCable channel that opens an interactive Net::SSH session with a requested PTY, pumps browser→stdin and channel→browser, and closes cleanly on disconnect (extend or sit alongside SshConnection).
  2. xterm.js UI: a "Console" action on each app's page that mounts a terminal, subscribes to the channel, and wires resize/keystroke events.
  3. Per-method command selection: resolve the remote command from app.deploy_method — kamal app exec, docker exec -it <container_name>, host shell / journalctl — reusing container_name, kamal_service, and app_dir.
  4. Authorization + audit + lifecycle: gate on webmaster/owner, record each session and command to a McpCall-style audit table, and enforce an idle session timeout that closes the PTY.

Acceptance criteria

  • From an app's page, an authorized operator can open a console and get an interactive prompt (e.g. a live rails console) in the browser.
  • Typed input is sent to the remote process and its output streams back in real time via the new channel.
  • A user without webmaster/owner role cannot open a console; the attempt is rejected.
  • Every console session and the commands run in it are recorded in the audit log.
  • An idle session is automatically closed after the timeout, tearing down the SSH PTY.

Dependencies & related

Builds on the existing SSH layer (app/services/ssh_connection.rb) and the ActionCable plumbing already used for deploy/script log streaming (config/routes.rb:138, app/channels/application_cable/). Reuses App deploy-method helpers (app/models/app.rb) and the McpCall audit pattern (app/models/mcp_call.rb, app/controllers/admin/mcp_calls_controller.rb). Related roadmap items: auto-deploy on push and rollbacks.
