Background worker management

Define, run, scale, and restart a managed app's background worker processes (Sidekiq and friends) from Conductor — the way Hatchbox does — instead of leaving them to manual SSH.

Pillar: Runtime Backends Priority: P1 Effort: M Status: Planned

Problem

Hatchbox treats background workers as first-class processes: you declare a worker (e.g. bundle exec sidekiq), pick a process count, and Hatchbox starts/stops/scales them and restarts them on every deploy. Conductor has no equivalent. A managed app today gets a web process and nothing else, so any Sidekiq, Solid Queue, or custom worker has to be wired up by hand over SSH and is invisible to the UI and to deploys. There is no model, no UI, and no MCP surface for an app's worker processes.

Current state

Conductor runs its own background jobs through Solid Queue inside Puma (SOLID_QUEUE_IN_PUMA), with recurring tasks declared in config/recurring.yml (SyncContainerStatusJob, RefreshServerMetricsJob, RunScheduledBackupsJob). That is Conductor's own runtime — it is unrelated to the worker processes of the managed apps it deploys.

For managed apps, deployment goes through three deployers in app/services/: native_deployer.rb, kamal_deployer.rb, and app_deployer.rb. On a native app's first deploy, NativeDeployer#deploy! runs a built-in app-setup script and then systemd-setup (app/services/native_deployer.rb:21). That systemd-setup script (seeded in db/seeds.rb:354) writes a single systemd unit whose ExecStart is bundle exec puma -C .../config/puma.rb (db/seeds.rb:377) — i.e. one web service per app and no worker unit at all. Kamal apps similarly get only the web role; ContainerStatus#sync_kamal (app/services/container_status.rb) detects liveness by the service=<kamal_service> label on the web container, so workers would not even be reflected in status.

All remote work is performed over SshConnection (app/services/ssh_connection.rb): execute for one-shots, execute_with_status for exit codes, and execute_stream for streamed output. Status is reconciled on a schedule by SyncContainerStatusJob calling ContainerStatus#sync!. There is no WorkerProcess model and nothing that generates worker units or roles.

Proposed design

Introduce per-app worker process definitions — each with a command, a process/replica count, and a target queue (and optional env). From those definitions, generate the right runtime artifacts per deploy method: for native apps, additional systemd units (one per worker, templated like the existing systemd-setup output but with the worker ExecStart and @-instance scaling); for kamal apps, a workers role (or accessory) in the Kamal config so Kamal manages worker containers alongside web. Lifecycle actions (start/stop/restart) run over SshConnection using systemctl (native) or kamal app boot/stop for the workers role (kamal). Workers are restarted automatically at the end of every deploy, and their status is surfaced through the existing status-sync pattern (extend ContainerStatus to also probe worker units / worker-labelled containers).

Implementation slices

  1. WorkerProcess model + surfaces. Add a WorkerProcess belonging to App (command, count, queue, enabled). Expose CRUD in the app UI (a Workers tab) and via an MCP tool so workers can be declared from the agent.
  2. Native: generate systemd worker units. Extend the native deploy path to template one systemd unit per worker (mirroring the systemd-setup seed at db/seeds.rb:354), using templated instances for count, then daemon-reload + enable over SshConnection.
  3. Kamal: map to a workers role. Emit a workers role (command + replicas + queue env) into the app's Kamal config so kamal_deployer.rb boots them with the web role.
  4. Status + restart-on-deploy. Restart workers as the final deploy step in each deployer, and extend ContainerStatus / SyncContainerStatusJob to report each worker's running state back into the UI.

Acceptance criteria

  • An operator can define a Sidekiq worker for an app (command, count, queue) from the UI and via MCP.
  • The worker actually runs on the box as a managed process — a systemd unit for native apps, a Kamal workers role for kamal apps.
  • Deploying the app restarts the workers automatically, with no manual SSH.
  • Worker run/stopped status is visible in the app's UI via the existing status-sync pattern.
  • Start / stop / restart actions on a worker take effect on the remote host.

Dependencies & related

Builds on the deployers (app/services/native_deployer.rb, kamal_deployer.rb, app_deployer.rb), the systemd-setup seed (db/seeds.rb), SshConnection (app/services/ssh_connection.rb), and the status-sync loop (ContainerStatus + SyncContainerStatusJob, config/recurring.yml).
