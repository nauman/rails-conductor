Reactive statuses everywhere (Turbo Streams)

Make statuses, deployments, and lists update live over Turbo Streams — no polling, no "Sync Status then refresh." Bring the whole UI up to the reactive standard the chat already meets.

Pillar: Fleet Control Priority: P1 Effort: M Status: In progress

Problem

Most of the UI is not reactive. App and container status badges require clicking Sync Status and then refreshing the page to see the result. Log tails (apps#logs, servers#logs) poll with a setInterval + fetch every few seconds. Deployments, script runs, and database pulls are pushed live, but through bespoke ActionCable channels + Stimulus DOM-patching rather than native Turbo Streams — more JS to maintain, and inconsistent with the rest of Hotwire. In 2026, on Rails 8 + Hotwire, "keep refreshing to see state" is the wrong default.

Current state

The foundation is all present: turbo-rails is pinned, ActionCable works, and the chat already does it right (turbo_stream_from + model broadcast_*). Slice 1 is shipped (2026-06-22): App#broadcast_status_badge pushes a Turbo Stream replace of the status badge whenever a status-relevant field changes, and apps/show subscribes via turbo_stream_from @app — the badge updates with no refresh. Deployments/script runs still use the custom deploy_channel/script_run_channel; index lists, the dashboard fleet view, and server pages remain manual; log tails still poll.

Proposed design

Push state from the model layer: App, Deployment, and Server broadcast targeted Turbo Stream replaces on status changes (the slice-1 pattern). Views subscribe with turbo_stream_from and render each badge/row inside a stable dom_id target. Fold the bespoke deploy/script-run channels into native Turbo Stream broadcasts (delete the hand-rolled DOM-patching JS). Finally, convert log tails from client polling to a server-side streamed append. Net effect: zero setInterval for state, far less custom JS, and consistent live updates everywhere.

Implementation slices

  1. Done (2026-06-22): live app status badge — App#broadcast_status_badge + turbo_stream_from on the app page.
  2. Live deployment status + list rows (deployments page and the app's deploy section) via Deployment broadcasts.
  3. Live badges on the apps index and the dashboard fleet view; Server status broadcasts on the servers pages.
  4. Retire the bespoke deploy_channel/script_run_channel DOM-patching into native Turbo Streams; convert the apps#logs/servers#logs polling into a streamed append.

Acceptance criteria

  • App, container, deployment, and server status update on screen with no manual refresh and no polling.
  • No setInterval remains for status; log tails arrive over a stream.
  • The hand-rolled status/DOM-patching JS is replaced by Turbo Stream broadcasts.
  • A second open tab reflects a status change within a second of it happening.

Dependencies & related

Uses turbo-rails + ActionCable (both already in place) and mirrors the chat's broadcast_*/turbo_stream_from pattern. Pairs naturally with live app log streaming (the log-tail slice) and makes deploy progress — including the self-deploy reconcile — visible without refresh.
