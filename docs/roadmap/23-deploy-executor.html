Deploy-executor rework

Stop running kamal inside the long-lived web container. Move each deploy into a short-lived, isolated executor so self-deploys don't kill their own job, secrets are injected at spawn (not rebuilt from the DB), and the runner is stateless.

Pillar: Runtime backends Priority: P1 Effort: L Status: Planned

Problem (the architecture inversion)

Kamal is a client-side / CI tool — meant to run from a laptop or CI job that orchestrates remote servers. Today Conductor runs kamal deploy from inside a long-lived app container that is itself a deploy target (KamalDeployer = "Conductor is the control machine"). That inversion is the root of every recurring deploy pain:

  • Self-reference: deploying Conductor means the deployer container is stopped mid-swap, killing the job before it records success. SelfDeployReconciler is a band-aid over a structural flaw.
  • Secret re-plumbing: kamal expects secrets in the runner's env, but the runner doesn't have the deploy-time secrets — so we rebuild .kamal/secrets from DB EnvVariables. The entire secret-juggling saga stems from this.
  • docker-over-SSH complexity: the container builds on the target daemon over SSH (host keys, known_hosts, stale identities).
  • Stateful multi-tenant runner: one container running many apps' deploys accumulates SSH/workspace cruft.

Current state

KamalDeployer runs kamal deploy as an in-process subprocess (LocalShell) inside the web container (jobs run there via SOLID_QUEUE_IN_PUMA). Worked around with the reconciler + master-key materialization + committed-secrets reuse for self-deploys. It works — but it's fragile and the wrong shape.

Proposed design

Separate control plane from deploy executor:

  • Control plane = the Conductor web app. Stores config, shows UI, exposes MCP. Never runs kamal in-process.
  • Deploy executor = a short-lived, isolated runner spawned per deploy — a one-off docker run --rm conductor-deployer … on the host daemon (or a dedicated worker), given the repo, env, and secrets at spawn, that runs kamal deploy and exits.

This makes self-deploy trivial (the executor outlives the web container's restart — no reconciler), removes the .kamal/secrets rebuild (secrets injected at spawn, ideally vault-resolved), and keeps each run stateless.

Grounded in prior art (researched 2026-06-24)

The real principle isn't "one-off container" — it's "the web app holds NO in-flight deploy state; Postgres does." The container is just the means.

  • kamal-ui & polaris-deploy both refuse to run server-side deploys (kamal-ui = read-only Docker observability; polaris = a macOS GUI shelling out to local kamal). The two closest comparables dodge this exact problem — confirmation it's hard, and that Conductor is the ambitious one.
  • ubicloud (Ruby/Postgres IaaS control plane) is the model: work is a DB row (Strand) run by a stateless worker; mutual exclusion is a DB lease (SELECT … FOR NO KEY UPDATE SKIP LOCKED + lease timestamp), so a crashed worker's lease just expires and the row is re-picked — crash-safe by construction. Secrets are pulled per-operation from encrypted columns, decrypted in-process at connect time, never written to a long-lived file. Every step is bounded + a watchdog kills wedged work.

Adopt: (1) persist structured deploy progress to the DB, not stdout — if the executor dies, stdout is gone (this was the "log says streaming but isn't" bug); (2) a deploy lease per app for one-at-a-time + auto-recovery; (3) use Solid Queue (already present) to spawn + supervise the executor — do NOT build ubicloud's custom scheduler.

Pitfall: re-entry ≠ idempotency. kamal deploy mid-swap is not idempotent — blindly retrying a torn deploy can double-act. Each phase must be check-then-act; decide per-phase whether to resume, roll back, or require a human.

Novel to us: none of the three ever redeploys the process orchestrating the deploy — the self-deploy / bootstrap-swap problem is ours alone, and externalizing the executor is the correct answer.

Implementation slices

  1. Extract a DeployRunner boundary + a deploy lease on the app (one deploy at a time; a crashed/expired lease is reclaimable). Package a conductor-deployer image (kamal + docker CLI + git).
  2. A Solid Queue job spawns the executor as a one-off container per deploy on the host daemon and supervises it; persist structured progress to the Deployment row (phase + status), not just streamed stdout; final status from the executor's exit, not the web job.
  3. Inject secrets at spawn (env or vault-resolved); delete the DB→.kamal/secrets rebuild path. Make each phase check-then-act so a crashed deploy is safely re-runnable. Self-deploy needs no reconciler.
  4. Retire SelfDeployReconciler (or keep as belt-and-suspenders); migrate the Docker + Native deployers onto the same runner boundary.

Acceptance criteria

  • A self-deploy completes and records success without a reconciler — the executor survives the web container's restart.
  • No deploy rebuilds .kamal/secrets from the DB; secrets are injected at spawn (or vault-resolved) and never persisted.
  • Each deploy runs in a fresh, stateless executor; no accumulated SSH/workspace state in the web container.
  • A crashed executor doesn't strand a deploy: its lease expires and the deploy is reclaimable/retryable, and status reflects DB-persisted progress (not lost stdout).
