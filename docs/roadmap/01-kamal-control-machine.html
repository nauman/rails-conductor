Deploy Kamal apps through Conductor (control machine)

Run kamal from Conductor's own container so managed Kamal apps deploy through Conductor — not just out-of-band from a laptop.

Pillar: Runtime Backends Priority: P0 Effort: M Status: In progress — code redesigned, infra pending

Problem

Conductor can trigger a Kamal deploy (the deploy_app MCP tool and apps#deploy dispatch to KamalDeployer via DeployAppJob), but the first version SSHed to the target server and ran kamal there. Proving the path against the live fleet box showed why that can't work:

fleet box 135.181.114.59:
  git: /usr/bin/git    docker: ok    buildx: ok    docker group: yes
  ruby:   MISSING      bundle: MISSING      kamal: MISSING

Kamal is a control-machine tool — the target only needs Docker. Running kamal on the box would die at kamal: command not found, and the box also has no git credentials to clone a private repo.

Current state

Redesign landed (commit a46b24e): KamalDeployer now treats Conductor's own container as the control machine. Conductor already bundles the kamal gem (Gemfile, kamal 2.10.1). The deployer:

  • checks out the app repo into a workspace (tmp/kamal/<slug> or KAMAL_WORKSPACE);
  • generates .kamal/secrets from the app's EnvVariables via KamalEnvWriter (Conductor = source of truth for secret values);
  • materializes the target server's SSH key (from SshKey#private_key) so Kamal's net-ssh can reach the box, and cleans it up afterward;
  • runs kamal deploy as a local subprocess via LocalShell (Open3, streamed to the deploy log), with Conductor's env injected (DEPLOY_SERVER_IP, registry creds, …).

Orchestration is unit-tested (test/services/kamal_deployer_test.rb, local_shell_test.rb) with an injected shell. What's not yet validated is the runtime environment inside Conductor's deployed container.

Proposed design — remaining infra

To make a real through-Conductor Kamal deploy succeed in production:

# config/deploy.yml (Conductor itself)
volumes:
  - "/var/run/docker.sock:/var/run/docker.sock"   # build/push from the container
  - "conductor_kamal_workspace:/rails/tmp/kamal"   # persist checkouts between deploys
  • Docker access: mount the host's /var/run/docker.sock so kamal can build/push (the host already has buildx). The deploy user is in the docker group.
  • kamal on PATH: confirm kamal resolves in the app checkout (it runs from the app dir; fall back to the app's bin/kamal or invoke via Conductor's bundle).
  • Repo auth: private repos need a deploy key / token at clone time — see GitHub App & deploy keys.
  • SSH to target: the materialized key must be wired into Kamal's net-ssh (deploy.yml ssh.keys / an ssh-config / ssh-agent). This is the main integration point to validate.
  • Registry: Conductor's container logs into Docker Hub using KAMAL_REGISTRY_PASSWORD from the app's env.

Implementation slices

  1. Done — control-machine redesign: LocalShell + KamalDeployer run kamal locally; generate secrets; materialize/cleanup the SSH key; stream logs. Unit-tested.
  2. Mount docker.sock + a persistent workspace volume into Conductor's container; redeploy Conductor.
  3. Wire the target SSH key into Kamal's net-ssh and confirm kamal resolves from the app checkout.
  4. Validate end-to-end: deploy kuickr via the deploy_app MCP tool; confirm the deploy log streams and the release boots; status-sync flips to running.
  5. Add a deployment_log MCP tool / API so deploy progress is observable over MCP (today only recent_logs reads ScriptRuns).

Acceptance criteria

  • Triggering deploy_app on a kamal app deploys its latest commit through Conductor, with live log streaming.
  • No ruby/kamal required on target servers — only Docker.
  • Secrets set in Conductor's UI (e.g. SECRET_KEY_BASE) reach the container; the target SSH key is materialized for the deploy and removed afterward.
  • A failed deploy marks the Deployment failed with the error in the log; a successful one flips status to running.

Dependencies & related

Builds on the Kamal-env bridge (KamalEnvWriter) and the kamal-aware status sync (both shipped). Unblocked private-repo deploys depend on GitHub App & deploy keys. Pairs with auto-deploy on push to deliver the full “push → deploy” loop.
