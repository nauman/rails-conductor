Deploy hooks (pre/post commands)

Let each app define ordered commands to run around its deploy — migrations beyond the defaults, cache warmups, asset syncs, notifications.

Pillar: Runtime Backends Priority: P2 Effort: S Status: Planned

Problem

Real apps need to run app-specific commands around a deploy: extra migration or seeding steps beyond the defaults, cache warmups, asset/CDN syncs, search-index rebuilds, or a notification to Slack when the release goes live. Conductor offers no place to hang any of this — the deploy is a fixed pipeline, so operators end up SSHing in by hand after every release, which defeats the point of a control plane.

Current state

Every deployer runs a hard-coded sequence with no user-defined extension points. NativeDeployer#deploy! (app/services/native_deployer.rb) runs a fixed run_script chain — on first deploy app-setup ("Setting up app directory") then systemd-setup ("Configuring systemd service"), and always app-deploy ("Deploying release"), followed by health_check. Each run_script loads a built-in Script by name, prepends env_exports, and streams output into the deploy log; there is no way to insert a command before or after any of these phases.

The Docker path (app/services/app_deployer.rb) is likewise a fixed steps array (ensure_docker → clone_or_pull_repo → build_image → stop_old_container → start_container → health_check → cleanup), and the Kamal path (app/services/kamal_deployer.rb) is just sync-repo + kamal deploy. App (app/models/app.rb) carries deploy config (deploy_method, branch, env_variables, health_check_path) but has no association for user hooks. Notably, Kamal apps do already support .kamal/hooks natively — but Conductor never generates them.

Proposed design

Add per-app, ordered deploy hooks, each with a phase (before_deploy / after_deploy / before_migrate), a command, and a run-on target (server — over SSH on the app server — or control — locally on Conductor). The deployer executes the relevant hooks between its existing steps. Storage can reuse the existing Script/ScriptRun machinery or a lighter dedicated Hook model belonging to App; given hooks are per-app and ordered, a dedicated Hook model is the cleaner fit, executed via the same SSH/LocalShell streaming the deployers already use so output lands in the deploy log. For Kamal apps, Conductor should generate .kamal/hooks scripts from the configured hooks rather than running them itself, so they run inside Kamal's own lifecycle — note the interplay so a hook isn't run twice.

Implementation slices

  1. Hook model (belongs_to :app; phase, command, run-on, position) with UI and MCP CRUD.
  2. Execute hooks in NativeDeployer at the right phases — before_deploy/before_migrate ahead of app-deploy, after_deploy after the health check passes — using the existing SSH streaming path.
  3. Surface hook output in the deploy log (same append_log/broadcast channel as built-in steps) so it's visible live and in history.
  4. Generate .kamal/hooks scripts for Kamal apps from the configured hooks (instead of Conductor-side execution), reconciling with Kamal's own hook lifecycle.

Acceptance criteria

  • A configured before_deploy hook runs before the release step, and a configured after_deploy hook runs after the health check — each at the correct phase.
  • Hook output appears inline in the deploy log alongside the built-in steps.
  • A failing hook aborts the deploy: the deployment is marked failed and subsequent steps do not run.
  • Hooks are creatable/editable/deletable from both the UI and an MCP tool, with stable ordering.
  • For Kamal apps, configured hooks materialize as .kamal/hooks scripts and are not also run twice by Conductor.

Dependencies & related

Touches the same deployer step sequences as the rollbacks plan (04-rollbacks.html) — both edit where NativeDeployer runs its phases. Reuses the SSH/streaming-log plumbing in app/services/native_deployer.rb and the Deployment#append_log/ActionCable broadcast in app/models/deployment.rb. The Kamal slice depends on the control-machine workspace generation in app/services/kamal_deployer.rb (alongside its .kamal/secrets writing). The MCP CRUD parallels the existing app-management tools.
