Multi-tenant MCP — "anyone can deploy"

Per-user / per-org MCP tokens scoped to an org's own apps, so any member (or their agent) can deploy their apps over MCP — not the whole fleet as admin.

Pillar: Agent-native Priority: P1 Effort: M Status: Shipped 2026-06-20

Shipped 2026-06-20 — complete

The MCP server accepts a per-user / per-org ApiToken bearer and runs the call as that user, scoped via the ActorScoped concern + a membership-checked resolve_organization. Delivered:

  • Org-bound scoping — a token is bound to one org (Current.organization); it can't see or act on another org's apps/servers even if the user belongs to both. Legacy CONDUCTOR_MCP_TOKEN keeps global admin scope.
  • Read-only vs deploy scope — ApiToken#scope (read/deploy); read-only tokens limited to fleet_status/recent_logs/deployment_log at the ToolRegistry choke point.
  • Self-serve issuance UI — the Tokens page (/mcp_tokens) mints/revokes org-scoped tokens with a scope picker; raw token shown once.
  • Privilege-escalation gaps closed across creation/admin tools (membership check, run_script scoping, admin-gated GitHub-App tools).

Covered by test/integration/multi_tenant_mcp_test.rb (9 cases). See the MCP guide.

Problem

Conductor's MCP server is live and fully drives deploys (see the MCP guide), but auth is single-tenant: there is one shared CONDUCTOR_MCP_TOKEN, and every call runs as User.admins.first. So "anyone can deploy" today means "anyone holding the root token, acting as admin across every org." That's fine for a single operator; it blocks handing MCP access to other members/customers, which is the point of the multi-tenant model already in the app (Organizations + Memberships + org-scoped /api/v1).

Current state

The MCP endpoint (app/controllers/mcp/server_controller.rb) authenticates a single env token and sets mcp_user = User.admins.first; tools run with that actor. Meanwhile the REST API (/api/v1) already does per-token, org-scoped auth (Api::V1::BaseController + a tokens resource) and the MCP audit log (McpCall) already records the affected org. The pieces exist; MCP just doesn't use per-user tokens yet.

Proposed design

Reuse the existing API-token model for MCP: a bearer token resolves to a user + org, and every tool call is authorized and scoped to that org's resources.

  • Token → actor. Look up the bearer in the API-token store (hashed), set mcp_user = that user and the active org = the token's org. Keep CONDUCTOR_MCP_TOKEN working as a legacy admin token (deprecation path).
  • Scope every tool. Resource tools (deploy_app, set_env_variable, update_app, provision_database, …) resolve apps/servers through the token's org only; cross-org access is denied, not silently global.
  • Per-token capabilities (optional). A read-only vs deploy scope on the token, so an org can hand out an inspect-only token.
  • Self-serve issuance. A "MCP tokens" section (mirroring the API tokens UI) to mint/revoke org-scoped tokens; show the connect snippet from the MCP guide.

Implementation slices

  1. MCP auth: resolve a bearer token to user+org via the API-token store; fall back to the legacy admin env token.
  2. Authorization: route every tool's resource lookup through the token's org; deny cross-org; cover with tests (extend the existing org-scoping specs).
  3. Issuance UI + revisions: mint/revoke org-scoped MCP tokens, read-only vs deploy scope, surfaced with the connect snippet.
  4. Docs: update the MCP guide — drop the "single shared token / admin" caveat once per-user tokens land.

Acceptance criteria

  • A non-admin member can mint an org-scoped MCP token and deploy their org's apps via MCP.
  • That token cannot see or act on another org's servers/apps/credentials.
  • A read-only token can call fleet_status/deployment_log but not deploy_app/set_env_variable.
  • The audit log attributes each call to the real user + org behind the token.

Dependencies & related

Builds on the existing org-scoped /api/v1 token auth and the McpCall audit log. Completes the agent-native pillar started by the live MCP server. Related: MCP guide, GitHub App (clone auth per org), and the org/membership model already in the app.
