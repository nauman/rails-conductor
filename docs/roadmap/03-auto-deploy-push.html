Auto-deploy on git push

Receive GitHub/GitLab push webhooks and deploy the matching app automatically — Hatchbox's signature push-to-deploy flow.

Pillar: Runtime Backends Priority: P0 Effort: M Status: Shipped 2026-06-20

Shipped 2026-06-20

Implemented as designed: apps gained auto_deploy + a generated webhook_secret; a public WebhooksController (POST /webhooks/:provider/:app_id) verifies the GitHub X-Hub-Signature-256 HMAC over the raw body, gates on auto_deploy + branch match, debounces in-progress deploys, and enqueues the existing DeployAppJob — no new deploy path. An "Auto-deploy on push" card on the app page toggles it and shows the payload URL + secret. Covered by test/integration/auto_deploy_webhook_test.rb (signature, auto_deploy gate, branch match, debounce, 404).

Problem

Conductor only deploys when a human pulls the trigger: a button in the UI or an explicit MCP call. Hatchbox's entire value proposition is the opposite — you git push and your app ships. Until Conductor reacts to pushes on its own, it can't honestly be sold as a Hatchbox replacement; every deploy still needs a person in the loop.

Current state

Deploys today have exactly two entry points, both manual:

  • Web UI — POST /apps/:id/deploy routed to AppsController#deploy (app/controllers/apps_controller.rb:47). It guards on @app.deployable? and on no in-progress deployment, then does deployment = @app.deployments.create!(user: current_user) and DeployAppJob.perform_later(deployment.id).
  • MCP tool — DeployAppTool#call (app/tools/deploy_app_tool.rb) resolves an app by id/name and runs the same deployments.create! + DeployAppJob.perform_later pair.

DeployAppJob (app/jobs/deploy_app_job.rb) is already backend-agnostic — it dispatches on app.deploy_method to NativeDeployer, KamalDeployer, or the default AppDeployer. So the deploy machinery is ready; what's missing is an automatic trigger.

There is no inbound webhook receiver anywhere in config/routes.rb (the closest thing is the mcp namespace and the API v1 namespace, neither of which accepts SCM push events). On the data side, App (app/models/app.rb) carries repository_url and branch but has no auto_deploy flag and no webhook_secret — there is nothing to verify a payload signature against and nothing to gate on.

Proposed design

Add a public WebhooksController that SCM providers post to. Each app gets a unique webhook_secret; the controller verifies the provider's HMAC signature header (e.g. X-Hub-Signature-256 for GitHub) against that secret, extracts the repo and ref from the payload, matches them to an App by repository_url + branch, and — if auto_deploy is on — enqueues the existing DeployAppJob. No new deploy path is introduced; we reuse the manual one.

# config/routes.rb
post "webhooks/:provider/:app_id", to: "webhooks#receive", as: :webhook

# app/controllers/webhooks_controller.rb
class WebhooksController < ActionController::Base
  skip_forgery_protection

  def receive
    app = App.find(params[:app_id])
    head :unauthorized and return unless valid_signature?(app)
    head :ok and return unless app.auto_deploy?
    head :ok and return unless ref_matches_branch?(app)

    return if app.deployments.in_progress.any? # debounce
    deployment = app.deployments.create!
    DeployAppJob.perform_later(deployment.id)
    head :accepted
  end

  private

  def valid_signature?(app)
    expected = "sha256=" + OpenSSL::HMAC.hexdigest(
      "SHA256", app.webhook_secret, request.raw_post)
    ActiveSupport::SecurityUtils.secure_compare(
      expected, request.headers["X-Hub-Signature-256"].to_s)
  end
end

The signature check must run against the raw request body, before parsing. Branch matching reads refs/heads/<branch> from the payload and compares to app.branch.presence || "main" (the same default both NativeDeployer and KamalDeployer already use).

Implementation slices

  1. Migration + model: add auto_deploy:boolean (default false) and webhook_secret:string to apps; generate the secret on create (alongside the existing generate_slug/generate_image_name before_validation hooks) and expose a "regenerate secret" action.
  2. WebhooksController#receive + route: provider-aware HMAC verification (GitHub X-Hub-Signature-256, GitLab X-Gitlab-Token), repo/branch extraction, and app matching by repository_url + branch.
  3. UI + MCP surface: an auto_deploy toggle and the webhook URL/secret on the app edit page (add :auto_deploy to app_params in AppsController); mirror the toggle in UpdateAppTool so agents can flip it.
  4. Concurrency/debounce: reuse the existing deployments.in_progress.any? guard and coalesce bursts of pushes so a rapid series of commits results in one deploy of the latest ref.

Acceptance criteria

  • A push to an app's configured branch, with a valid signature, creates a Deployment and enqueues DeployAppJob within seconds.
  • A request with a missing or wrong signature is rejected with 401 and never enqueues a deploy.
  • With auto_deploy = false, a valid signed push is acknowledged (200) but triggers no deployment.
  • A push to a non-matching branch is ignored.
  • While a deployment is in progress, additional pushes do not stack up duplicate concurrent deploys.

Dependencies & related

Closely paired with GitHub App & deploy keys: the same GitHub App can both deliver these webhooks (and supply the shared secret) and authorize the private-repo clone that the resulting deploy performs, plus post commit-status updates back. Builds directly on the existing DeployAppJob dispatch (app/jobs/deploy_app_job.rb) and the Deployment lifecycle in app/models/deployment.rb; no deployer changes required.
