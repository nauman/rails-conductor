Rollbacks & release history

Give every deploy method a retained release history and a one-click (or one-tool) rollback to the previous good version.

Pillar: Runtime Backends Priority: P1 Effort: M Status: Planned

Problem

When a deploy ships a bad release, there is currently no way to get back to the version that was running a minute ago. The operator's only recourse is to push a revert commit and run a full fresh deploy — slow, and useless when the repo is fine but the release is broken (bad migration, bad asset build, bad env). Hatchbox, the tool Conductor aims to replace, retains a stack of releases and offers one-click rollback. Conductor needs parity here before it can be trusted with production fleets.

Current state

Conductor records deploys but keeps no rollback-able artifact. app/models/deployment.rb persists status, log, timing (started_at/completed_at) and a commit_sha column, with a STATUSES lifecycle (pending → building → deploying → succeeded/failed) and a successful scope — but there is no release version, no retained build, and no rollback action anywhere on the model.

The native path is destructive by design: NativeDeployer#deploy! (app/services/native_deployer.rb) runs the built-in app-deploy script in-place, and the analogous Docker path (app/services/app_deployer.rb) does git fetch origin && git reset --hard origin/#{app.branch} straight into a single app_dir (/opt/conductor/apps/#{slug}). There is one working tree and one image tag (#{app.image_name}:latest) — the prior release is overwritten, so there is nothing to roll back to.

The Kamal path is the exception worth calling out: KamalDeployer#deploy! (app/services/kamal_deployer.rb) shells out to kamal deploy, and Kamal itself keeps prior versioned container images on the target host. So for kamal? apps the release history effectively already exists on the host — Conductor just never surfaces or invokes it. The deployer routing lives in app/jobs/deploy_app_job.rb, which switches on app.deploy_method (native / kamal / else Docker), so any rollback dispatch must branch the same way.

Proposed design

Treat rollback as a per-deploy_method capability, recording a release identifier on every successful Deployment so history is queryable regardless of backend.

  • Kamal apps: lean on Kamal's existing versioned images. Rollback dispatches kamal rollback [version] from the same checkout/control-machine setup KamalDeployer already builds (workspace, SSH key, secrets), targeting the version recorded on the chosen prior Deployment.
  • Native apps: adopt a Capistrano-style layout — deploy into timestamped releases/<version> directories with a current symlink, instead of resetting one working tree in place. Rollback = repoint current at the prior release dir and restart the service. Requires reworking the built-in app-deploy/app-setup scripts to be release-dir aware.
  • Shared: store the released version/ref on Deployment (extend the existing commit_sha with a release_version), expose a Rollback action in the UI on the app's deployment history, and add a rollback_app MCP tool mirroring the existing deploy_app tool so agents can roll back too.

Implementation slices

  1. Record a release version/ref per Deployment on success (column + populated by each deployer; Kamal stores the Kamal version, native stores the timestamped release id).
  2. Native release-dir support: rework deploy scripts to build into releases/<ts> and switch the current symlink atomically; restart systemd service against current.
  3. Rollback action dispatching by deploy_method — UI button on deployment history plus a rollback_app MCP tool, routed like DeployAppJob (native → symlink switch, kamal → kamal rollback).
  4. Prune old releases (keep N most recent native release dirs; document Kamal's own retention).

Acceptance criteria

  • Triggering a rollback on a native app restores the previously running release (symlink repointed, service restarted) without a fresh git pull/build.
  • Triggering a rollback on a Kamal app invokes kamal rollback against the prior version and the previous container is serving traffic again.
  • Each successful deploy records a release version/ref, and the app's deployment history shows the rollback-able releases in order.
  • Rollback is invokable from both the UI and the rollback_app MCP tool.
  • Old native releases are pruned to a bounded count.

Dependencies & related

Builds on the deployer routing in app/jobs/deploy_app_job.rb and the Deployment lifecycle in app/models/deployment.rb. The native slice requires reworking the built-in app-deploy/app-setup scripts that NativeDeployer#run_script loads. The Kamal slice depends on the control-machine setup in app/services/kamal_deployer.rb (see plan-kamal-control-machine.html). The MCP tool parallels the existing deploy_app tool, and the per-app deploy-hooks plan (plan-deploy-hooks.html) shares the same deployer step-sequence touchpoints.
