In-container task runner (db:seed / rake / migrate)

Run a one-off, non-interactive task inside a deployed app's container — db:seed, a rake task, a data migration, a maintenance script — from the UI and over MCP, with streamed output.

Pillar: Agent-native Priority: P1 Effort: M Status: Planned

Problem

Conductor can deploy an app and run host-level provisioning scripts, but it has no way to run an app-level task in the running container. Anything like db:seed, a backfill rake task, a one-off data fix, or db:migrate out of band requires an operator to drop to a terminal and run kamal app exec --reuse "bin/rails …" with the right env. That breaks the "manage the fleet from the UI / via an agent" promise and — critically — means an AI agent driving Conductor over MCP cannot finish jobs that need a task run (it surfaced when a deploy needed db:seed and there was no tool for it).

Current state

The closest tool is run_script (app/tools/run_script_tool.rb), which runs a predefined provisioning Script on a server (host shell), not a command in an app container. KamalDeployer (app/services/kamal_deployer.rb) already knows how to reach the target and run kamal with the app's env from Conductor's container, and App carries deploy_method (kamal/docker/native) plus container_name / kamal_service / app_dir helpers — so the addressing exists; only the exec entrypoint is missing. This is the non-interactive sibling of the web console (interactive PTY); they can share the command-resolution layer.

Proposed design

Add a TaskRun (modeled on ScriptRun/Deployment) that runs one command in the app container and streams output over ActionCable, plus an MCP tool run_app_task. The remote command is resolved per deploy_method: kamal app exec --reuse "<cmd>" for Kamal apps, docker exec <container_name> <cmd> for docker/native. Tasks are chosen from a small allow-list of safe verbs (db:seed, db:migrate, rails runner, named rake tasks) rather than arbitrary shell, every run is recorded for audit (like McpCall), and access is gated to webmaster/owner.

Implementation slices

  1. TaskRun model + job that execs one command in the app container via Conductor's control-machine path, streaming stdout/stderr to the deployment-log UI.
  2. Per-method command resolution (kamal app exec vs docker exec) reusing App deploy-method helpers.
  3. MCP tool run_app_task (allow-listed verbs: db:seed, db:migrate, rake, rails runner) + a UI "Run task" action on the app page.
  4. Authorization + audit: gate to webmaster/owner, record each task run; redact secrets in output like the MCP audit log.

Acceptance criteria

  • From an app's page (and over MCP), an authorized operator can run db:seed / a rake task in the running container and watch output stream live.
  • The same tool works for Kamal and docker/native apps via per-method command resolution.
  • Only allow-listed verbs run; arbitrary shell is not exposed through this tool.
  • Every task run is recorded in an audit log; a non-privileged user cannot run tasks.

Dependencies & related

Built on the Kamal control machine exec path and the ActionCable log-streaming used by deploys/script runs. Powers seed management (which calls db:seed through it). Interactive counterpart: web console.
