Backup restore + verification (R2)

Close the loop on backups: restore a Postgres backup from R2 into a target database, and verify backups are actually restorable — a backup you can't restore isn't a backup.

Pillar: Data & backups Priority: P0 Effort: M Status: Planned

Problem

Conductor creates and schedules pg_dump backups to R2, but there is no restore path and no verification — so backups are untested and recovery means manual pg_restore gymnastics under pressure. This is the single riskiest gap in the Data pillar.

Current state

Backup creation + scheduling to R2 works (docs/plans/backups-r2.md, postgres-restore.md notes the restore flow doesn't exist). Postgres clusters + per-app databases are provisioned via PostgresClusterClient. No restore, no verify, no PITR.

Proposed design

A guarded restore service: pull a chosen backup from R2 → restore into a target (a fresh database, a scratch verify DB, or — with explicit confirmation — an existing one). Verification runs restores into a throwaway DB on a schedule and records pass/fail, so every backup has a "last verified" state. Destructive restores are gated (confirm + audit) and exposed as a deliberate MCP tool, not arbitrary shell.

Implementation slices

  1. BackupRestore service: fetch from R2 → pg_restore into a fresh/scratch database, streamed log (like deploys).
  2. Verification job: scheduled restore-into-throwaway-DB per backup; record "last verified" + size/row sanity checks.
  3. UI: per-backup Restore + Verify actions, restore history, and a red/green "restorable" badge.
  4. Gated restore-into-existing (confirm + audit) + MCP conductor_backup tool (restore/verify, allow-listed, never arbitrary).

Acceptance criteria

  • An operator can restore any R2 backup into a fresh database from UI and MCP, watching streamed output.
  • Scheduled verification marks each backup restorable / not, surfaced as a badge + alert on failure.
  • Restoring over an existing database requires explicit confirmation and is audited.

Dependencies & related

Builds on the Postgres cluster lifecycle + R2 integration (the backup store). Failure surfacing feeds alerts and Continuous maintenance.
